## Context

当前命令型工作流可以快速创建变更提案，但在 PPT 生成场景中，生成任务常被一次性描述，导致单代理提示词负担过重，输出在结构与视觉维度不稳定。为降低认知耦合，需要将任务拆分为专职角色，并通过显式上下游协议进行串联。

## Goals / Non-Goals

**Goals:**
- 定义一个可复用的 agent team 编排模型，覆盖 4 类产物：大纲、逐页文案、背景、布局。
- 明确每个代理的输入、输出与质量检查项。
- 在 `/opsx:propose` 的指令中加入可执行的团队分工与顺序。
- 保持与现有 OpenSpec 产物流程兼容，不破坏 `proposal/design/specs/tasks` 生成链路。

**Non-Goals:**
- 本次不实现新的图像生成模型接入。
- 本次不定义最终渲染引擎内部坐标算法。
- 本次不覆盖动画编排与转场效果优化。

## Decisions

1. 采用“编排者 + 4 专家代理”模型。
- 编排者负责拆分任务、传递上下文、聚合结果和回退。
- 4 个专家代理只关注单一职责，降低提示词复杂度。
- 备选方案：单代理多阶段提示。放弃原因：上下文拥挤，质量波动更大。

2. 执行顺序采用“串并结合”。
- 阶段 1：OutlineAgent 产出页面结构。
- 阶段 2：CopyAgent 基于大纲生成每页文案。
- 阶段 3：BackgroundAgent 与 LayoutAgent 并行，输入为大纲 + 文案。
- 阶段 4：Composer 汇总为每页完整稿（文本 + 背景 + 布局）。
- 备选方案：严格串行 4 阶段。放弃原因：延迟更高且背景/布局相互依赖较弱。

3. 定义统一中间数据结构。
- SlideOutline[]: 页级主题、目标、受众、时长。
- SlideCopy[]: 标题、副标题、要点、备注。
- SlideVisual[]: 背景主题、配色、视觉元素约束。
- SlideLayout[]: 版式模板、区块坐标、层级与对齐。
- 选择结构化 JSON 作为代理间契约，避免自然语言歧义。

4. 引入轻量质量门禁。
- 页数一致：outline/copy/layout 必须同页数。
- 文案完整：每页至少有标题与 2 个要点（可配置）。
- 可读性：背景与文本对比满足阈值规则。
- 可渲染：布局区块不重叠、边界不越界。

## Risks / Trade-offs

- [Risk] 代理间契约过严导致创意受限 -> Mitigation: 保留可选字段与 `notes` 扩展。
- [Risk] 并行阶段输出风格不一致 -> Mitigation: 统一主题 token（语气、色板、行业风格）由编排者下发。
- [Risk] 新增编排逻辑增加维护成本 -> Mitigation: 保持角色接口最小化并补充示例。

## Migration Plan

1. 在 `/opsx:propose` 说明中加入 agent team 章节和执行顺序。
2. 为生成类需求添加“启用 team 模式”的触发条件（关键词或显式要求）。
3. 输出结构化产物到 proposal/design/specs/tasks，确保 apply 阶段可消费。
4. 预留回退策略：若 team 模式失败，降级为单代理并记录缺失项。

Rollback:
- 回滚为现有单代理流程，禁用 team 模式触发逻辑，不影响 OpenSpec 文件结构。

## Open Questions

- 质量门禁阈值（如对比度）是否需要按主题类型差异化？
